# 이해민 의원실 통합 플랫폼 최종 PRD

작성일: 2026-04-15
상태: 개발 착수 확정본
프로젝트 코드명: `haimin-issue-desk`

---

## 1. 한 줄 정의

이 제품은 **이해민 의원실 / 과방위 업무를 정리하는 내부 이슈 데스크**를 중심으로, 그 위에 **공개 의정 브리프**와 **정책 제안 접수**를 얹는 **하나의 제품군**이다.

핵심은 “예쁜 시민용 사이트”가 아니라, **의원실이 현안을 놓치지 않고 정리하고 설명할 수 있게 만드는 업무 시스템**이다.

---

## 2. 이번 프로젝트의 최종 결론

### 채택
- 하나의 제품군으로 만든다.
- 하나의 코드베이스, 하나의 DB, 하나의 관리자 시스템으로 간다.
- 사용자 표면은 `desk`, `brief`, `voice`로 나눈다.
- MVP의 중심축은 **내부 이슈 데스크(`/desk`)**다.
- 공개 사이트는 **의정 브리프(`/brief`)**를 먼저 연다.
- `voice`는 코드상 구현하되 **초기 배포에서는 feature flag로 OFF** 한다.

### 기각
- 3개 프로젝트를 서로 다른 서비스/레포로 분리하는 안
- `voice`를 먼저 여는 안
- 시민 의견을 공개 게시판처럼 보여주는 안
- AI가 생성한 텍스트를 사람 승인 없이 자동 발행하는 안
- 의원 점수화/랭킹/정치성 강한 비교 지표를 전면에 두는 안
- 공개 댓글, 공개 토론, 공개 신고 첨부파일 업로드를 v1에 넣는 안
- `국회 과방위에 바란다`를 공식 청원처럼 보이게 만드는 안

## 3. 제품 구조

### 3-1. 내부 모듈: `/desk`
의원실 실무자가 사용하는 운영 콘솔.

역할:
- 과방위 현안 수집
- 출처 정리
- 기사/자료 중복 제거
- 이슈 카드 생성
- 이슈별 브리프 작성
- 발의안/표결/회의/발언 연결
- 후속조치 기록
- 공개 브리프 발행 승인
- `voice` 접수 triage

### 3-2. 공개 모듈: `/brief`
시민/언론/이해관계자가 보는 공개 브리프.

역할:
- 이해민 의원 소개
- 과방위 핵심 이슈 브리프
- 이해민 의원 의정활동 타임라인
- 발의안/표결/회의/발언의 구조화된 공개
- 출처와 갱신 시각 표시

### 3-3. 공개 모듈: `/voice`
정책 제안 및 현장 의견 접수.

최종 포지셔닝:
- 공식 청원 아님
- 공식 민원 아님
- 의원실 참고용 정책 제안/현장 의견 접수

초기 상태:
- 개발은 한다
- 운영 배포에서는 OFF
- 관리자 화면에서 켤 수 있게 한다

## 4. 사용자와 핵심 문제

### 4-1. 1차 사용자
- 하우준 및 의원실 실무자

핵심 문제:
- 현안, 보도자료, 회의, 발의안, 기사, 후속조치가 흩어져 있다.
- 과방위 이슈를 매일 쫓아가며 손으로 정리해야 한다.
- 의원님께 설명할 자료와 외부에 보여줄 자료가 분리돼 있다.
- 공식 데이터는 많지만 바로 쓰기 어렵다.

### 4-2. 2차 사용자
- 이해민 의원실을 보는 시민, 기자, 업계 관계자

핵심 문제:
- 의원실이 무엇을 하고 있는지 한눈에 보기 어렵다.
- 국회 데이터는 있지만 너무 분산돼 있다.
- 현안별 입장과 후속조치를 추적하기 어렵다.

### 4-3. 3차 사용자
- 정책 제안자 / 업계 관계자 / 제보자

핵심 문제:
- 과방위 현안에 대해 구조적으로 의견을 남길 창구가 없다.
- 남겨도 의원실 업무 흐름에 연결되기 어렵다.

## 5. MVP 우선순위 고정

### 1순위: Desk
이번 개발의 본체다.

반드시 포함:
- 로그인
- 역할 기반 권한
- 소스 수집 파이프라인
- 이슈 카드 CRUD
- 태그/분류/상태 관리
- 출처 중복 제거
- 공개 브리프 초안 작성/수정/승인/발행
- 데이터 freshness 표시

### 2순위: Brief
Desk에서 승인된 결과를 공개 노출하는 레이어.

반드시 포함:
- 홈
- 이해민 의원 소개
- 최신 브리프 목록
- 브리프 상세
- 활동 타임라인
- 출처 표시
- 마지막 갱신 시각

### 3순위: Voice
개발은 하되, 배포는 feature flag OFF.

반드시 포함:
- 의견 유형 선택
- 익명/가명 제출 기본
- 연락처 선택 입력
- 별도 동의 UI
- Turnstile
- rate limit
- 관리자 triage

## 7. 정보 구조 최종안

### 홈
- 의원실 소개
- 최근 브리프 3~5개
- 주요 과방위 카테고리
- 데이터 출처 안내
- “공식 청원 아님” 안내는 홈이 아니라 `voice`에서만 노출

### 브리프
- 필터: 카테고리 / 날짜 / 키워드
- 카드 요소: 제목, 3줄 요약, 관련 키워드, 관련 법안 수, 마지막 갱신 시각
- 상세: 배경, 현재 상황, 이해민 의원/의원실 관련 활동, 관련 법안/회의/발언, 출처 목록

### 활동 타임라인
- 자동 데이터: 발의안, 표결, 위원회 일정, 회의록 연결, 발언 링크
- 수동 데이터: 의원실 후속조치, 보도자료, 현장 방문, 간담회 메모
- 타임라인은 “사실 기록” 위주로 구성하고, 정치적 수사/홍보 문구는 최소화

### Desk 대시보드
- 오늘 새로 들어온 소스 수
- 검토 대기 이슈 수
- 발행 대기 브리프 수
- 최근 7일 카테고리별 소스 추이
- 수동 동기화 버튼

## 8. 카테고리 체계 최종안

카테고리는 **관리자 수정 가능**으로 설계한다.
초기 seed는 아래 11개로 넣는다.

1. AI·데이터·고영향 AI
2. 데이터센터·AIDC·전력·PPA
3. 사이버보안·침해사고·정보보호
4. 통신·이동통신·망 이용
5. 방송·미디어·플랫폼 규제
6. OTT·콘텐츠·저작권
7. 반도체·국가전략기술
8. 우주·과학기술·연구인프라
9. R&D 예산·거버넌스
10. 개인정보·디지털권리·딥페이크
11. 규제기관·법안소위·과방위 운영

중요:
- 이 11개는 고정 진리가 아니다.
- 운영 중 변경 가능해야 한다.
- 모든 수집/브리프/활동/의견은 다중 카테고리를 가질 수 있다.

## 10. 자동화와 AI 최종안

### 채택
- 자동 수집
- 자동 중복 제거
- 자동 태그 추천
- 자동 초안 생성(선택)

### 금지
- AI 자동 발행
- AI가 작성한 시민 의견 답변 자동 전송
- 시민 의견/내부 메모를 별도 검토 없이 외부 AI에 전송

### 운영 원칙
- v1은 **LLM 없이도 완전히 동작**해야 한다.
- LLM 연동은 `AI_ENABLED=false`여도 앱 전체가 정상 작동해야 한다.
- LLM은 “도우미”일 뿐, 워크플로우의 필수 요소가 아니다.

### v1의 AI 사용 범위
- 공개 출처 기사/자료의 3줄 요약 초안
- 카테고리 추천
- 중복 문서 유사도 보조
- 브리프 초안 문장 정리 보조

## 11. 로그인/권한 최종안

### 공개 영역
- 로그인 없음

### 관리자 영역
- 이메일+비밀번호 기반 로그인
- 역할 4종:
  - `admin`
  - `editor`
  - `reviewer`
  - `viewer`

### 운영 원칙
- 소셜 로그인 없음 (v1)
- 관리자 계정은 수동 생성
- 첫 배포 시 seed 계정 2개 생성
- 프로덕션 전환 전 TOTP 2단계 인증 추가 예정

이유:
- v1에서 카카오/네이버 소셜 로그인까지 넣을 필요가 없음
- 내부 사용자 수가 적음
- 외부 서비스 의존성과 개인정보 범위를 줄여야 함

## 12. `voice` 최종 정책

### 제품명
- 최종 명칭: **과방위 정책 제안**
- 보조 설명: “이해민 의원실이 검토를 위해 받는 정책 제안 및 현장 의견”

### 절대 금지
- “국회 청원”처럼 보이게 하기
- “국민동의청원”과 혼동되는 카피 쓰기
- 공개 게시판처럼 운영하기

### 제출 유형
1. 정책 제안
2. 산업·현장 의견
3. 개인 민원
4. 협업/인터뷰/자료 전달

### v1 폼 정책
- 익명 또는 가명 제출 기본
- 이름: 선택
- 이메일: 선택
- 휴대전화: 없음
- 주소: 없음
- 주민번호류: 없음
- 첨부파일: 없음

### 민원 분기
- “개인 민원”은 제출 직후 안내문으로 별도 연락 채널을 우선 제시
- `voice`는 정책 제안용이지 민원 처리 시스템이 아님

### 보관 정책 (v1.5 이후 적용)
- 무연락 익명 제안: 180일 후 삭제
- 연락처 포함 제안: 1년 후 삭제 또는 익명화
- 보안 로그(IP 원문 등): 7일 보관 후 파기/해시화

### 공개 정책
- 시민 제출 내용은 공개하지 않음
- 담당자만 열람
- 외부 발행 시에는 개별 의견 원문이 아니라 “종합 의견” 형태로만 반영

## 13. 선거/정치 리스크 대응 최종안

### 운영 원칙
- 공개 페이지는 “의정활동 기록/브리프”로 설계한다.
- 문구는 사실 중심으로 간다.
- 선거운동성 문구, 경쟁 후보 비교, 지지 호소 카피 금지.

### 제품 기능
- `election_mode` 전역 플래그
- election mode ON일 때:
  - 자동 발행 금지
  - reviewer 승인 필수
  - 홈 상단 배너 문구 단순화
  - CTA 최소화
  - `voice` 홍보 배너 비노출

### 현재 정책
- 2026-04-15 기준 운영 환경을 고려하여 election mode를 기본 ON으로 출발한다.

## 16. 인프라 최종안

## 20. 운영 정책 최종안

### 발행 정책
- 초안 작성자와 발행 승인자는 다를 수 있다.
- 공개 브리프는 reviewer 승인 후에만 공개된다.
- source가 하나도 없는 브리프는 발행할 수 없다.

### 소스 정책
- 원문이 사라질 수 있으므로 제목/요약/링크/수집시각은 저장한다.
- 저작권 문제를 피하기 위해 기사 전문 전체 공개는 하지 않는다.
- 외부 기사 본문은 내부 검토용 저장 범위를 최소화한다.

### 로그 정책
- 관리자 주요 행위는 모두 audit log에 남긴다.
- 비밀번호/민감 원문은 로그에 남기지 않는다.

## 25. 최종 선언

이 프로젝트는 이제 더 이상 리서치 단계가 아니다.

**최종 제품 정의는 다음과 같다.**

> 하나의 제품군, 하나의 코드베이스, 하나의 데이터 모델.
> 내부 본체는 과방위 이슈 데스크.
> 공개 레이어는 의정 브리프.
> 의견 접수는 후순위이자 feature-flagged 모듈.

이 문서를 기준으로 바로 개발에 들어간다.
